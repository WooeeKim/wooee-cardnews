# ux-writing-kr (말맛 / malmat) — 한국어 UX 마이크로카피

화면에 박히는 한 줄을 잘 쓰는 스킬이다. 버튼 글자, 에러 한 줄, 빈 화면 문구처럼 **짧지만 사용자 행동을 좌우하는 말**을 한국어 감각으로 뽑고 다듬는다.

영어권 UX 라이팅 도구는 한국어에서 무너진다. 한국어 UI 카피의 어색함은 대부분 **영어 번역투**("~하실 수 있습니다", "성공적으로 완료되었습니다")와 **높임 체계 흔들림**(한 화면에서 해요체와 합쇼체가 섞임), **말맛 부재**(딱딱하고 사무적)에서 나온다. 이 스킬은 그 세 가지를 한국어 네이티브로 잡는다.

이 스킬의 규칙은 즉흥이 아니라 **정통 UX 라이팅 리서치를 한국어에 맞게 옮긴 것**이다. Nielsen Norman Group의 에러 메시지·마이크로카피 연구, Kinneret Yifrah의 《Microcopy: The Complete Guide》, Material Design·Apple HIG의 라이팅 가이드, Mailchimp·Shopify Polaris·GOV.UK의 콘텐츠 스타일 가이드가 바탕이다. 이 원리들이 한국어 표면·높임·번역투에 어떻게 적용되는지는 [`references/foundations.md`](references/foundations.md)에 정리돼 있다.

## 무엇보다: 기능 정확성이 카피보다 먼저다

카피는 인터페이스의 약속이다. "삭제됩니다"라고 썼는데 실제로는 보관함으로 이동한다면, 멋진 문구라도 거짓말이다. **버튼이 실제로 무슨 동작을 하는지, 에러가 실제로 왜 났는지 모르면 추측하지 말고 사용자에게 되묻는다.** 모르는 상태로 그럴듯한 문구를 지어내지 않는다.

## 그다음: 안심(reassurance)을 설계한다

좋은 마이크로카피의 숨은 일은 **사용자의 불안을 낮추는 것**이다. NN/g 연구가 반복해 보여주듯, 사람들이 이탈하는 지점은 대개 "지금 무슨 일이 일어났지?", "이거 눌러도 되나?", "내 데이터·돈은 괜찮나?" 같은 **불확실성의 순간**이다. 카피는 그 불안을 줄이는 가장 싼 도구다.

불안이 솟는 4대 순간과 안심 처방:

- **대기/처리 중** — 침묵은 불안이다. 무슨 일이 진행 중이고 얼마나 걸리는지 알린다. "잠시만요" + 진행 상태. ("처리 중"만 던지고 끝내지 않는다.)
- **에러** — 사용자를 탓하지 않고, 원인 + 다음 행동 + (가능하면) "당신 잘못이 아니에요" 신호. 막다른 길을 만들지 않는다.
- **되돌릴 수 없는 행동(삭제·전송·결제)** — 결과를 분명히 알리고, 가능하면 실행 취소(undo)나 확인 한 단계를 둔다. "되돌릴 수 없어요"를 미리 말해주는 게 친절이다.
- **돈·개인정보** — 무엇이·누구에게·어떻게 되는지 투명하게. 모호함은 곧 불신. "받는 사람만 볼 수 있어요" 같은 한 줄이 전환을 살린다.

안심은 장황함이 아니다. **정확히 필요한 한 줄**이 과한 설명보다 더 안심시킨다.

## 두 가지 모드

요청을 보고 자동으로 판단한다.

- **생성 모드** — "이 버튼 뭐라고 쓸까", "토스트 문구 뽑아줘"처럼 새 문구가 필요할 때. 맥락을 받아 변형안 3~5개 + 추천 1개를 낸다.
- **비평·개선 모드** — 기존 문구를 주며 "이거 어색해", "다듬어줘"라고 할 때. 문제를 짚고(번역투·높임 흔들림·길이·모호함 등) 대안을 낸다.

## 입력 받기 (맥락 체크리스트)

좋은 마이크로카피는 맥락 없이는 못 나온다. 다음이 안 주어졌으면 **추측하지 말고 짧게 물어본다.** 단, 사용자가 빨리 보고 싶어 하면 합리적 가정을 1줄로 밝히고 진행한 뒤 가정이 틀리면 고치겠다고 한다.

- **표면(surface)**: 버튼인가 에러인가 토스트인가. 표면마다 길이·말투 규칙이 다르다. → `references/surfaces.md`
- **동작/상황**: 이 버튼을 누르면 실제로 뭐가 되는가. 이 에러는 왜 났는가. (정확성의 핵심)
- **제품 보이스**: 친근(반말 가까운 해요체) / 정중(합쇼체) / 위트(장난기). 명시 없으면 주변 카피나 서비스 성격으로 추정하고 1줄로 밝힌다.
- **길이 제약**: 버튼 글자 수, 한 줄 토스트 등. 모바일은 특히 짧아야 한다.
- **금지어/표기 규칙**: 쓰면 안 되는 단어(예: 어떤 서비스는 "결제" 대신 "송금"). 있으면 반드시 지킨다.
- **높임 수준**: 보통 보이스에서 따라오지만, 한 화면 안에서 **절대 섞이면 안 된다.**

## 출력 형식

생성 모드는 항상 이 형태로 낸다. 변형은 **서로 다른 전략**이어야 한다(짧은 버전, 다정한 버전, 행동 강조 버전 등). 그냥 단어만 바꾼 동의어 나열은 쓸모없다.

```
맥락 ▸ {표면} / 보이스 {친근|정중|위트} / 길이 {제약} / 금지어 {있으면}

추천 ▸ {가장 좋은 한 줄}
대안 ▸ {다른 전략의 한 줄}
대안 ▸ {또 다른 전략의 한 줄}
대안 ▸ {필요시 1~2개 더}
이유 ▸ {왜 추천이 나은지 한 줄. 버린 표현이 왜 별로인지도 짧게.}
```

비평·개선 모드는 이렇게 낸다.

```
원본 ▸ {받은 문구}
진단 ▸ {문제 1~3개. 카테고리로: 번역투 / 높임 흔들림 / 너무 김 / 모호함 / 책임 회피체 / 과장 등}
추천 ▸ {고친 한 줄}
대안 ▸ {다른 방향 한 줄}
이유 ▸ {한 줄}
```

여러 표면을 한 번에 요청받으면(예: "이 모달의 제목·본문·버튼 다"), 표면별로 묶어서 위 형식을 반복한다.

## 품질 렌즈 3겹

뽑거나 고친 모든 문구는 머릿속에서 이 3겹을 통과시킨다. 충돌하면 **(1) 정확성 → (2) 명확성 → (3) 안심 → (4) 말맛** 순으로 우선한다. 예쁘지만 헷갈리는 문구보다, 덜 예뻐도 분명하고 안심되는 문구가 낫다.

### 1. UX 원칙 (짧고 분명하게)

- **사용자 행동 중심 동사.** "확인" 같은 막연한 말보다 그 버튼이 하는 일을 ("저장", "보내기", "삭제"). 단 너무 길어지면 맥락상 자명한 선에서 줄인다.
- **군더더기 제거.** "~하실 수 있습니다", "~하기 위해서는", "주시기 바랍니다"는 대개 잘라도 된다.
- **긍정·능동 프레이밍.** "실패했습니다"보다 다음 행동을 ("다시 시도"). 사용자를 탓하는 말투 금지("잘못 입력하셨습니다" → "이메일을 다시 확인해 주세요").
- **모바일 길이 감각.** 버튼은 보통 2~6자, 토스트는 한 줄. 길면 잘리거나 안 읽힌다.

### 2. 번역투·AI티 제거 (한국어 네이티브 감각)

영어 UI를 직역한 티가 가장 흔한 어색함이다. 대표 패턴과 치환은 `references/translationese-lens.md`에 정리돼 있다. 핵심만:

- "성공적으로 ~되었습니다" → "{무엇}을 {했어요/완료}" (success 직역 제거)
- "~하실 수 있습니다" 남발 → 능동 청유 ("~해 보세요", "~하기")
- "오류가 발생했습니다" → 원인 + 다음 행동 ("연결이 끊겼어요. 다시 시도")
- "귀하 / 본인 / 사용자님" 같은 번역체 호칭 → 보통 생략하거나 자연스러운 한국어 호칭

### 3. 말맛 (한국 서비스 결)

토스·배민·당근 같은 서비스가 잘하는 건 **사람이 옆에서 말해주는 느낌**이다. 사무적 통보가 아니라 대화. 다만 보이스에 맞춰야 하고(은행 앱이 배민처럼 굴면 안 됨), 과한 애교나 유행어는 금세 촌스러워지니 절제한다. 레지스터·보이스·말맛 디테일은 `references/voice-register.md`.

## 주의 사항

- **정확성이 최우선.** 동작·결과를 모르면 지어내지 말고 묻는다.
- **금지어는 무조건 지킨다.** 사용자가 "결제 쓰지 마"라고 하면 변형 전부에서 뺀다.
- **한 화면 높임 일관.** 해요체로 시작했으면 끝까지 해요체.
- **과장·거짓 금지.** "최고의", "단 한 번의 클릭으로" 같은 마케팅 과장은 UI 카피에 넣지 않는다(마케팅 카피와 다름).
- **변형은 전략이 달라야 한다.** 동의어 교체 말고 접근 자체를 바꾼다.
- **이 스킬은 짧은 인터페이스 문구 전용.** 긴 글 윤문이나 광고 장문 카피 요청이면 그렇게 말하고 범위를 좁힌다.
